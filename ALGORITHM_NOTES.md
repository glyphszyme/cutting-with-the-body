# 가로/세로 알고리즘 개선 노트

## 현재 상황

### 현실 공간
- show 페이지를 맥 미니에서 데스크톱으로 들어간 후, 천장의 빔 프로젝터가 아래로 사영
- 200X80(cm) 크기의 요가 매트가 바닥에 있음
  - 격자 하나의 크기가 2X2(cm)
  - 세로 30개, 가로 90개
  - 즉, 실제 격자판의 크기는 180X60(cm)임.
  - 매트 내 격자판은 .container와 일치함

### adjust (모바일)
- 사실상 핸드폰을 컨트롤러로 바꿔주는 역할
- 세로형 모바일 웹
- 가로, 세로 수치를 직접적으로 보여주기보다는
단 하나의 사각형만으로 시각적 피드백을 전달함
- 사각형 위의 좌우 스크롤이 가로의 감소/증가와 대응됨

### show (데스크톱)
- 핸드폰으로 접속한 adjust 웹에 따라 변형되는 다소 수동적인 웹
- 세로형(중요하지 않음) 데스크톱 웹
- 현실에서는 이 웹을 180X60(cm) 면적에 빔 프로젝터로 송출함
- 사영된 글자 하나는 2cm여야 하기에 내가 상수 C를 하드코딩하여 조절하여야 함
    - 왜? 빔의 해상도, 사영면까지의 거리가 아직 정해지지 않았으므로
    - W_show = √(키×어깨) × C × width(메인 파라미터)
    - H_show = √(키×어깨) × C × height(W_show에 따라 결정됨)

## 문제점

1.
사용자가 키, 어깨너비, 가로, 세로를 입력하면, 대략적인 "실제 가로, 세로" 길이가 나옴

2.
우선 정확하게 재고, 이것을 반올림, 버림 뭐 하든 알아서 2cm 격자 개수로 변환함

3.
가로와 세로의 최소 개수는 1개가 되어도 됨.
하지만 그러면 각각 세로, 가로가 엄청나게 늘어나는데, 그게 격자판을 초과하면 안 됨

4.
최악 = {
    adjust에서 show로 데이터를 보내고, 그걸 바탕으로 계산하고
    다시 adjust로 back해서 컨트롤러 사각형의 레이아웃을 결정
}

5.
adjust 자체에서 데이터를 받은 순간,
show에서 출력 가능한 가로,세로 range를 알아내야함

6.
그럼에도 동시에 adjust와 show의 사각형 비율은 같아야 함
15:17과 16:17처럼 미묘하게 다른 건 의미가 없음

7.
사각형 길이가 극단적인 것을 걱정하는 것은
show나 adjust나 모두 동일함


## 해결 방안

1.
adjust -> show로 단 한 번의 데이터가 전송되면 좋을 듯
하지만 왔다갔다 하는 게 더 빠르다면 그렇게 해도 됨

2.
show는 현실에서 출력되는 레이아웃이기 때문에 정말 엄격해야 함
